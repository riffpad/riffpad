# AI-Native 轻量化代码灵感草稿本 (Riffpad) 产品需求文档 (PRD)

## 1. 导言与产品定位

### 1.1 背景与痛点
当前 AI 辅助软件开发（AI Coding）市场存在显著的断层：
* **传统 LLM 网页版（如 ChatGPT / Claude）：** 属于“纯文本对话流”。头脑风暴时的代码片段散落在漫长对话中，无法直接运行、难以沉淀，重新回顾（Recap）的心智成本极高。
* **重型 AI IDE（如 Cursor / Windsurf）：** 属于“重型生产力工具”。要求用户必须处于严肃开发状态（开电脑、建目录、配置本地环境、管理 Git）。

这导致人类 80% 诞生于通勤、散步、会议等非电脑前的**碎片化、转瞬即逝的灵感**，因沉淀阻力过重而胎死腹中。

### 1.2 产品核心定位
本产品定位为 **“AI 时代的代码灵感草稿本 (AI-Native Sketchbook)”**。它是一个追求极致轻量、极速启动、随时随地可用的有状态工作区。
* **它不是什么：** 它不是用来写工业级、可直接上线生产环境代码的 IDE。
* **它是什么：** 它是整个 AI 软件工程链路的**最上游孵化器**。核心任务是“低摩擦捕捉灵感 -> 跑通原型验证 -> 一键桥接下游（Cursor/Claude Code等）”。

### 1.3 核心价值主张 (Value Propositions)
1. **零摩擦 (Zero Friction)：** 像打开手机便签一样快，无需起名，无需配置环境，开箱即用。
2. **可执行的思考空间 (Runnable Sandbox)：** 每个工作区内置轻量沙箱与自主 Agent，能直接读写文件、执行 Bash 命令，瞬间实现想法可视化。
3. **完美的桥接能力 (The Upstream Bridge)：** 验证成功的“草稿”可一键无缝重构并打包，生成下游 IDE 直接可用的高质量项目骨架。

## 3. 功能需求详情 (Functional Requirements)

### 3.1 极速启动与零摩擦工作区 (Instant Workspace)
* **需求描述：** 用户进入应用的核心目标是记录，任何阻碍其开始输入的步骤都是流失点。
* **关键特性：**
  * **闪击一笔 (New Spark)：** 登录后默认首页即为输入框。点击创建新项目时，系统必须在 500ms 内分配并启动全新的有状态沙箱，严禁弹出“请选择语言、请选择框架、请输入项目名称”的传统弹窗。
  * **随机代号机制：** 系统为未命名的项目自动赋予富有美感的随机代号（如 `amber-spark-9`、`velvet-breeze-42`），降低用户的认知摩擦。
  * **有状态目录树：** 告别纯文字聊天流。工作区左侧为自动生成的轻量级目录树（由 Agent 自动管理文件），右侧为灵感画布/预览窗，下方为意图交互区。

### 3.2 智能 Agent 与轻量沙箱 (Smart Agent & Sandbox)
* **需求描述：** 赋予 AI 真正操作环境的能力，让“灵感”能够跑起来。
* **关键特性：**
  * **自主代理操作：** 内置 AI Agent 拥有该沙箱的完整读写权限。它不仅是回复文本，而是能直接理解用户的模糊意图，在沙箱内创建、修改、重构文件。
  * **Bash 闭环执行：** Agent 能够自主在沙箱内运行 Bash 命令（如 `pip install` 轻量依赖、运行 Python 脚本、启动 Mock Server）。执行过程默认对用户隐藏，但保留实时日志流供高级用户查看。
  * **预览优先画布 (Preview-First)：** 针对前端、Web、数据可视化相关的灵感，沙箱内置静态/动态 Web 服务代理。右侧画布可一键切换为“实时预览”模式，支持渲染 HTML/JS、Tailwind CSS、React/Vue 单页组件，让用户直观看到原型的动态反馈。

### 3.3 下游生态桥接器 (Downstream Bridge)
* **需求描述：** 作为“上游孵化器”，必须完美解决“草稿”向“正规军开发”的过渡。
* **关键特性：**
  * **AI 骨架重构：** 当用户结束头脑风暴，点击“推向开发”时，Agent 会对沙箱内的乱麻代码进行一键重构。
  * **标准上下文打包：** 导出包内不仅包含规范的代码骨架，还会强制生成一份高品质的 `README.md`（内容包含：该点子的产品核心逻辑、已验证可行的技术路径、使用的第三方 API 说明、AI 提示词上下文描述包）。
  * **一键集成：** 支持一键推送到临时 GitHub 仓库，或生成一个专属的 Zip 压缩包，让用户在电脑端打开 Cursor 的瞬间，能完全继承在手机/平板上发生过的所有思考。

## 4. 非功能性需求 (Non-Functional Requirements)

### 4.1 性能与响应 (Latency & Startup)
* **容器预热机制：** 后台必须维护一个处于预热状态的轻量级虚拟化环境池（如 Firecracker VMM 或极简 Docker 池），确保用户点击“New Spark”时是直接分配而非冷启动，冷启动感知时间必须 `< 1秒`。
* **极致的双端适配 (Mobile-First)：** UI 必须深度适配移动端与平板。
* **高精度语音输入：** 移动端交互区必须深度集成语音转文字（如 Whisper 模型）。因为在走路、通勤等场景下，语音口述想法是捕捉灵感最高效的手段。

### 4.2 安全与内核隔离 (Security & Isolation)
* **运行期内核隔离：** 由于 Agent 具有执行任意 Bash 命令的权限，必须防止恶意 Prompt 越狱攻击宿主机。沙箱必须处于严格的内核级隔离环境中。
* **资源配额限制 (Quota)：** 限制单个沙箱的网络带宽、CPU 及内存使用额度，防止死循环脚本耗尽服务器资源。

### 4.3 生命周期与存储管理 (Lifecycle & Cost)
* **自动休眠机制：** 作为“草稿本”，默认项目具有生命周期。未标记长期保存的临时沙箱，在非活跃状态 48 小时后自动休眠，释放服务器算力。
* **结构化文本保留：** 沙箱休眠时仅保留结构化的文本快照与对话历史，再次唤醒时（感知时间 `< 3秒`）重新恢复沙箱现场，确保 SaaS 运营的算力成本绝对可控。

## 5. 关键业务指标 (KPIs) 与验证标准

1. **激活转化率：** 用户从进入首页到成功创建第一个沙箱并触发 AI 修改文件的比例（目标 > 70%）。
2. **启动延迟：** 核心指标 “New Spark 响应时间”（目标 P95 < 800ms）。
3. **灵感流失率：** 用户创建沙箱后，由于等待时间或交互复杂而选择直接关闭 App 的比例（目标 < 5%）。
4. **桥接率：** 用户使用“一键导出 / 推送到 GitHub / 桥接 Cursor”功能的频次占总活跃项目数的比例（用以验证“上游工具”的定位是否成立）。
